# Architecture & Design Document: TBEP (Telegram Bot for English Practice)

## 1. Overview

This document describes the technical architecture and implementation design for TBEP v1: a Python-based Telegram bot using webhooks, built around a class-based command/message processing pipeline, and integrating with an LLM (initially ChatGPT) to drive persona-based conversation and grammar corrections.

This document assumes familiarity with the Product Requirements Document (PRD_TBEP.md) and focuses purely on implementation and architecture.

---

## 3. High-Level Architecture

```
Telegram servers
      │
      │  HTTPS webhook (POST update)
      ▼
 main.py (webhook server / PTB Application)
      │
      ├── Command Router ──► CommandProcessor subclasses
      │                         (StartCommandProcessor, TopicCommandProcessor,
      │                          ProfileCommandProcessor, HelpCommandProcessor,
      │                          ResetCommandProcessor)
      │
      └── Message Router ──► MessageProcessor subclasses
                                (TextMessageProcessor)
                                      │
                                      ├──► LLMClientFactory
                                      │        │
                                      │        ▼
                                      │   LLMClient subclass
                                      │   (ChatGPTClient)
                                      │
                                      └──► UserStateStore (in-memory)
```

**Flow summary:**
1. Telegram sends an update to the bot's webhook endpoint.
2. `python-telegram-bot`'s `Application` dispatches the update to the appropriate handler based on whether it's a command or a plain text message.
3. Commands are routed to the matching `CommandProcessor` subclass; plain text messages are routed to `TextMessageProcessor` (the only `MessageProcessor` implementation in v1).
4. `TextMessageProcessor` reads/updates per-user state, obtains an `LLMClient` via `LLMClientFactory`, and calls `send()` to get the persona reply and correction.
5. The processor returns a response (or responses) that the handler sends back to the user via Telegram.

## 5. Class Design

### 5.1 `CommandProcessor` (abstract base class)

Responsible for handling a single Telegram command (`/start`, `/profile`, `/topic`, `/help`, `/reset`).

```python
from abc import ABC, abstractmethod

class CommandProcessor(ABC):
    def __init__(self, user_state_store):
        self.user_state_store = user_state_store

    @abstractmethod
    def process(self, user_id: int, args: str) -> str:
        """
        Process the command for the given user.
        `args` is the raw text following the command (e.g., the persona
        description for /profile, or empty string for /help).
        Returns the response text to send back to the user.
        """
        raise NotImplementedError
```

**Subclasses (one per command):**

- `StartCommandProcessor` — applies default persona/topic if unset, resets conversation, generates and returns an in-character opening message.
- `ProfileCommandProcessor` — sets the persona from `args` and returns a confirmation without changing conversation history.
- `TopicCommandProcessor` — sets the topic from `args` and returns a confirmation without changing conversation history.
- `HelpCommandProcessor` — returns static help text describing commands and how corrections work.
- `ResetCommandProcessor` — clears conversation history only (keeps persona/topic) and returns a confirmation.

Only `StartCommandProcessor` triggers an LLM call to produce an opening message. Profile and topic commands only update user state, while reset only clears history.

### 5.2 `MessageProcessor` (abstract base class)

Responsible for handling a non-command message from the user.

```python
from abc import ABC, abstractmethod

class MessageProcessor(ABC):
    def __init__(self, user_state_store, llm_client_factory, **llm_config):
        self.user_state_store = user_state_store
        self.llm_client = llm_client_factory.create(**llm_config)

    @abstractmethod
    def process(self, user_id: int, content) -> dict:
        """
        Process the incoming message content for the given user.
        Returns a dict with the response(s), e.g.:
          {"persona_reply": "...", "correction": "..." or None}
        """
        raise NotImplementedError
```

Rather than holding a reference to the factory and calling it on every `process()` invocation, the `MessageProcessor` base class calls `llm_client_factory.create(...)` once, in its constructor, and stores the resulting `LLMClient` instance as `self.llm_client`. This keeps `process()` simpler (no per-call factory lookup) at the cost of needing a new `MessageProcessor` instance if the LLM provider needs to change at runtime — an acceptable tradeoff for v1, since there's only one provider anyway.

**Subclass (v1):**

- `TextMessageProcessor` — the only implementation in v1. It:
  1. Retrieves the user's current state (persona, topic, conversation history) from `UserStateStore`.
  2. Uses `self.llm_client` (already instantiated by the base class constructor via `LLMClientFactory`) to get the persona reply.
  3. Calls `self.llm_client.send(...)` again (or a single structured call — implementation detail) to get the correction.
  4. Appends the turn to conversation history in `UserStateStore`.
  5. Returns a dict containing the persona reply and the correction (or `None` if no correction is needed).

### 5.3 `LLMClient` (abstract base class)

```python
from abc import ABC, abstractmethod

class LLMClient(ABC):
    def __init__(self, api_key: str, model: str = None, **config):
        self.api_key = api_key
        self.model = model
        self.config = config

    @abstractmethod
    def send(self, messages: list) -> str:
        """
        Sends a list of messages (conversation history + system prompt)
        to the LLM and returns the text response.
        """
        raise NotImplementedError
```

**Subclass (v1):**

- `ChatGPTClient(LLMClient)` — implements `send()` using the OpenAI ChatGPT API. Constructor receives the API key (and optionally model name, temperature, etc.) via config, loaded from environment variables in `config.py`.

### 5.4 `LLMClientFactory`

```python
class LLMClientFactory:
    @staticmethod
    def create(provider: str = "chatgpt", **config) -> LLMClient:
        if provider == "chatgpt":
            return ChatGPTClient(**config)
        raise ValueError(f"Unsupported LLM provider: {provider}")
```

- Defaults to returning `ChatGPTClient` when no provider is specified.
- `TextMessageProcessor` and `StartCommandProcessor` obtain their `LLMClient` instance through this factory rather than instantiating a concrete class directly — this is the extension point for adding more providers later (e.g., Claude, Gemini) without changing calling code.

### 5.5 `UserStateStore` (abstract base class) / `UserStateStoreMemory` (implementation)

Not explicitly requested at first, but needed to support the design — following the same base-class-plus-implementation pattern as `CommandProcessor`, `MessageProcessor`, and `LLMClient`.

```python
from abc import ABC, abstractmethod

class UserStateStore(ABC):
    @abstractmethod
    def get(self, user_id: int) -> "UserState":
        raise NotImplementedError

    @abstractmethod
    def set_persona(self, user_id: int, persona: str):
        raise NotImplementedError

    @abstractmethod
    def set_topic(self, user_id: int, topic: str):
        raise NotImplementedError

    @abstractmethod
    def reset_history(self, user_id: int):
        raise NotImplementedError

    @abstractmethod
    def append_turn(self, user_id: int, role: str, content: str):
        raise NotImplementedError
```

**Subclass (v1):**

- `UserStateStoreMemory(UserStateStore)` — the only implementation in v1, backed by a plain in-memory dictionary keyed by Telegram user ID.

```python
class UserStateStoreMemory(UserStateStore):
    def __init__(self):
        self._state = {}  # user_id -> UserState

    def get(self, user_id: int) -> "UserState":
        ...

    def set_persona(self, user_id: int, persona: str):
        ...

    def set_topic(self, user_id: int, topic: str):
        ...

    def reset_history(self, user_id: int):
        ...

    def append_turn(self, user_id: int, role: str, content: str):
        ...
```

`UserState` holds: `persona: str`, `topic: str`, `history: list[dict]` (role/content pairs for LLM context).

**Note on persistence:** `UserStateStoreMemory` is in-memory only for v1 — data is lost on process restart/redeploy. This is an accepted tradeoff for simplicity at this stage. A persistent store (e.g., `UserStateStoreSQL`) can be introduced later as another `UserStateStore` subclass without changing calling code.

## 6. Tooling: `uv`, `ruff`, `mypy`

- **`uv`** manages the virtual environment and dependencies for the project. `pyproject.toml` declares project metadata and dependencies; `uv.lock` pins exact resolved versions. Common commands: `uv sync` (install/sync environment), `uv add <package>` (add a dependency), `uv run <command>` (run a command inside the managed environment, e.g. `uv run pytest`).
- **`ruff`** handles both linting and formatting, configured via a `[tool.ruff]` section in `pyproject.toml`. Run via `uv run ruff check .` (lint) and `uv run ruff format .` (format).
- **`mypy`** performs static type checking, configured via a `[tool.mypy]` section in `pyproject.toml`. Run via `uv run mypy src`. Type hints should be used throughout `src/` (function signatures, class attributes) to get meaningful coverage from `mypy`.
- **`pytest`** is the test runner, configured via a `[tool.pytest.ini_options]` section in `pyproject.toml` (e.g., `testpaths = ["tests"]`). Run via `uv run pytest`.
- All four tools are expected to be run locally during development and are natural candidates for CI checks later (CI pipeline specifics are out of scope for this document — see Section 11).

## 7. Command & Message Routing (`main.py`)

`main.py` is responsible for:
1. Loading configuration (bot token, LLM API key, webhook URL/port) from environment variables via `config.py`.
2. Constructing shared singletons: `UserStateStoreMemory` (the `UserStateStore` implementation used in v1), `LLMClientFactory`.
3. Constructing the `python-telegram-bot` `Application` and registering handlers:
   - `CommandHandler("start", ...)` → `StartCommandProcessor`
   - `CommandHandler("profile", ...)` → `ProfileCommandProcessor`
   - `CommandHandler("topic", ...)` → `TopicCommandProcessor`
   - `CommandHandler("help", ...)` → `HelpCommandProcessor`
   - `CommandHandler("reset", ...)` → `ResetCommandProcessor`
   - `MessageHandler(filters.TEXT & ~filters.COMMAND, ...)` → `TextMessageProcessor`
4. Starting the application in webhook mode (`Application.run_webhook(...)`), binding to the port Railway provides and setting the public webhook URL with Telegram.

Each handler function is a thin adapter: it extracts `user_id`/`args`/`content` from the Telegram `Update`, delegates to the relevant processor's `process()` method, and sends the returned text (or texts) back via `update.message.reply_text(...)` — sending the persona reply and correction as two separate messages, per the PRD.

## 8. Webhook & Deployment (Railway, Free Tier)

- The bot runs `python-telegram-bot`'s built-in webhook server (`Application.run_webhook`), which starts an internal HTTP server bound to the port Railway assigns via the `PORT` environment variable.
- On startup, the app registers its public webhook URL with Telegram (`https://<railway-app-domain>/<secret-path>`), using Telegram Bot API's `setWebhook`.
- A secret path/token in the webhook URL is used to prevent unauthorized requests from reaching the handler (basic security measure, since the endpoint is public).
- **Free tier considerations:**
  - Railway's free tier may sleep/restart the service after inactivity — combined with in-memory state, this means user state (persona/topic/history) can reset unexpectedly. This is accepted for v1 given the in-memory storage decision.
  - No persistent volume is used in v1 since there's no DB to persist.
- Environment variables required: `TELEGRAM_BOT_TOKEN`, `OPENAI_API_KEY`, `WEBHOOK_SECRET_PATH`, `PORT` (provided by Railway).

## 9. Error Handling

- All `LLMClient.send()` calls are wrapped in try/except within `TextMessageProcessor` and `StartCommandProcessor`.
- On any LLM error/timeout, the processor returns the fixed fallback text: **"An error occurred. Try again in a moment."** — sent as the only response for that turn (no correction message in this case).
- Telegram API errors when sending messages are logged; `python-telegram-bot`'s built-in error handler (`Application.add_error_handler`) is used to catch and log unhandled exceptions across the app.

## 10. Extensibility Notes

- **New LLM providers:** Add a new `LLMClient` subclass (e.g., `ClaudeClient`) and extend `LLMClientFactory.create()` to support a new `provider` value. No changes needed in `MessageProcessor`/`CommandProcessor` classes.
- **New commands:** Add a new `CommandProcessor` subclass and register a `CommandHandler` in `main.py`.
- **New message types (e.g., voice):** Add a new `MessageProcessor` subclass (e.g., `VoiceMessageProcessor`) and register a corresponding handler — the abstract base class design already anticipates this.
- **Persistent state:** Add a new `UserStateStore` subclass (e.g., `UserStateStoreSQL`) backed by a database, and use it in place of `UserStateStoreMemory` when constructing shared singletons in `main.py`. No changes needed in processors, since they depend only on the `UserStateStore` interface.

## 11. Testing

Unit tests are in scope for this project. The `tests/` folder mirrors the `src/` folder structure (see Section 4), with one test module per source module. Key areas to cover:

- `CommandProcessor` subclasses — correct state transitions (persona/topic set, history reset), with the `LLMClient` mocked for `/start`.
- `MessageProcessor` (`TextMessageProcessor`) — correct handling of persona reply + correction generation, correction omitted when none needed, fallback message returned on simulated LLM errors, with the `LLMClient` mocked.
- `LLMClient` (`ChatGPTClient`) — request construction and response parsing, with the underlying OpenAI API call mocked.
- `LLMClientFactory` — correct provider resolution (defaults to `chatgpt`; raises on unsupported provider).
- `UserStateStore` (`UserStateStoreMemory`) — get/set persona, get/set topic, history reset, and turn appending behave correctly per user.

`pytest` is the test runner (see Section 6), configured via `pyproject.toml`. Specific fixtures, coverage thresholds, and CI integration are left as implementation details beyond this document.

## 12. Coding Guidelines

- **Methods should be small and do one thing.** Each method should have a single, clear responsibility that's easy to describe in one sentence.
- If a method starts doing multiple things (e.g., validating input, then transforming data, then calling an external service, then formatting a response), those parts should be extracted into their own small private/helper methods, called by the original outer method. The outer method becomes a short, readable sequence of calls to well-named helpers, rather than one long block mixing multiple concerns.
- This applies throughout the codebase, including (but not limited to):
  - `CommandProcessor.process()` implementations — e.g., separate "resolve persona/topic," "reset state," and "generate opening message" into distinct helper methods rather than one large `process()` body.
  - `TextMessageProcessor.process()` — e.g., separate "build LLM prompt," "call LLM for persona reply," "call LLM for correction," and "update history" into distinct helper methods.
  - `LLMClient` implementations — e.g., separate "build request payload," "call the API," and "parse the response" rather than combining all three in `send()`.
- This guideline exists purely to keep the code readable and maintainable; it's a stylistic/structural principle rather than a hard architectural constraint, and should be applied with judgment rather than as a strict line-count rule.
